## 1. 总览：Option A → B 转向要点 + 修复映射

**Option A 现状（要废）**：被控端 `FileTreeState::snapshot_rows`（`filetree.rs:348`）每帧抽可见行指纹，变化即整发 `FileTreeSnapshot`（`main.rs:1022-1031`）；控制端缓存进 `RemoteWs.remote_filetree`（字段 `remote_ws.rs:159`，类型 `RemoteFileTree{root,rows}` `remote_ws.rs:184-189`），`show_remote`（`filetree.rs:1843`）按 `rows+depth` 自绘扁平树；点击产 `FileTreeOp` 经 `mod.rs:736`→`main.rs:6795 send_filetree_op` 回传，被控端 `apply_remote_filetree_op`（`main.rs:1063`）执行展开/折叠/cd/打开。**展开态、cwd 节奏、懒加载全耦合进被控端渲染** → 6 问根因。

**Option B 转向**：
- 控制端自持「展开态 + 目录缓存 + 请求号/代次」状态机，**展开/折叠纯本地不发帧**。
- 控制端展开未缓存目录 → 发 `ListDir` → 被控端 `read_dir_worker`（`filetree.rs:1675`）直接读盘回 `ListDirResult`（脱离被控端渲染/快照节奏）。
- 控制端渲染**写一个远程孪生 `add_remote_node`**（ltreeview 路径），复用 ltreeview 的 closer 三角；不再自绘扁平行。
- 被控端只服务无状态原语：`ListDir`（读目录）、`Fetch`（读文件，带 ACK 背压）、`Put`（写文件）、`MkDir`（建目录）；所有递归/打开/复制粘贴由控制端编排。

**修复映射**：

| 问题 | 根因（A） | 修法（B） | 主改动 |
|---|---|---|---|
| #1 三角图标错 | 自绘 `▾/▸`（`filetree.rs:1903`）+ `is_open` 默认态坑 | 复用 ltreeview closer 三角（`add_node` `NodeBuilder::dir` `filetree.rs:1503`） | 控制端渲染换 ltreeview 孪生 |
| #2 不显示文件 | 快照只含已展开目录的可见行；根未展开则空 | 控制端按 `ListDirResult.entries`（含 `is_dir=false`）渲染 File 分支（照 `filetree.rs:1446`） | 同上 |
| #3 无法展开子目录 | 依赖被控端 `remote_expand`+渲染触发懒加载（`filetree.rs:414`），树栏不可见时不渲染 | 控制端点开 → 本地入队 `ListDir`，与被控端渲染无关 | 控制端状态机 |
| #4 cd 后刷新延迟数秒 | 双击=Cd→等被控端 OSC9;9→换根→重发快照整条慢链（`filetree.rs:1908-1911`） | 双击目录=本地展开（不 cd）；根来源改被控端 `RootChanged` 轻量帧主动推 | 双击语义 + 新帧 |
| #5 在被控端打开 | `apply_remote_filetree_op` Open 在被控端 `open_with_default`（`main.rs:1084`） | 删此分支；控制端 `Fetch` 字节→落临时文件→控制端本地 `open_with_default`（`filetree.rs:2077`） | 删 Open + Fetch 流程 |
| #6 展开态被同步 | 展开/折叠经 `FileTreeOp::Expand/Collapse` 回传被控端 | 完全不发；展开态只存控制端 `tree` | 删发送、被控端不再处理 |

## 3. 被控端服务 handler（`crates/lumen-app/src/main.rs` + `remote_ws.rs`）

被控端只在 `pump_remote`（`main.rs:956`，`controlled` 分支 `988-1035`）排空 pending 队列，读/写盘**一律后台线程 + 回包**（绝不在 pump 帧内同步 IO，照 `ensure_listing` 后台线程模式 `filetree.rs:1658`，根由注释 `filetree.rs:1654-1657`：慢速网络盘同步 read_dir 会冻结整个应用）。

**复用点与改造**：

- **`ListDir` → `read_dir_worker` 的 `show_hidden` 变体**。当前 `read_dir_worker`（`filetree.rs:1675`）无条件 `is_hidden` 过滤（`:1694`）。新增签名 `read_dir_worker_ex(dir: &Path, show_hidden: bool)`，`show_hidden` 时跳过 `is_hidden`；本地树调用方传 `false` 保持现状（或令旧 `read_dir_worker` 转调 `_ex(dir, false)`）。已含：目录在前排序（`:1700`）、`MAX_ENTRIES_PER_DIR` 截断 + overflow（`:1701`）、`ENUM_HARD_CAP` 封顶（`:1687`）、读失败 `Err(())`（`:1680`）。逐项 `(PathBuf, bool)` 转 `DirEntry{ path: p.display().to_string(), name: display_name(&p)（filetree.rs:1800）, is_dir }`。`Err(())` → `ListDirResult{ err: Some(FsErr::Io), entries: vec![], overflow: 0, path }`（细分 `NotFound/PermissionDenied` 按 `io::ErrorKind` 留片4；片2 先粗粒度 `Io`）。

- **`Fetch` → 后台线程分块读，ACK 窗口背压**。`spawn_fetch`：`File::open` + `metadata().len()`；`len > FETCH_MAX_LEN` → `FileErr{TooLarge}`。否则 `FileBegin{total_len}` → 循环读 `FILE_CHUNK` 发 `FileChunk{seq, data}`，**但已发未 ACK 的块数不得超 `FETCH_WINDOW`**：worker 需感知 ACK。架构上 worker 是独立 `thread::spawn`、出站经 `cmd_tx`、ACK 经 `apply_relay` 进主线程——worker 看不到 ACK。**解法**：被控端 Fetch 不用「一发到底」的纯 worker，而是在 `RemoteWs` 维护 `inflight_fetch_src: HashMap<u64, FetchSrcJob>`（持 `File` 句柄 + `next_seq` + `acked_seq`）；worker 仅做首帧 `metadata`，之后每收一个 `FileChunkAck` 在 `apply_relay` 里读并发出窗口内的下一块（读盘在主线程？否——读盘仍须后台）。**落地选择**（实现时二选一，记 §10.D 旁注）：
  - (a) **有界 channel 背压**：把 Fetch worker 的出站改走一个**有界** `sync_channel(FETCH_WINDOW)`，worker `send` 阻塞即天然背压，主线程 ACK 推进消费——但当前出站统一走无界 `cmd_tx`，需为 Fetch 单开有界中转，复杂。
  - (b) **ACK 驱动逐块读（推荐）**：worker 只发 `FileBegin`；`inflight_fetch_src` 记 `File`；收到 `FetchReq` 后主线程立即触发「读并发首 `FETCH_WINDOW` 块」——读盘放一次性 `spawn` 里把前 W 块读好缓冲，之后每个 `FileChunkAck` 触发再 `spawn` 读下一块。简化版：每文件一个专属 worker + 一个 `mpsc::Receiver<()>`「许可」通道，主线程收 ACK 即 `permit_tx.send(())`，worker `permit_rx.recv()` 阻塞领许可再读下一块。**此法 worker 自背压、主线程零读盘、内存恒为 W 块**。终稿采 (b)。
  - 空文件：`FileBegin{total_len:0}` + `FileEnd`（无 chunk）。读错 `FileErr{err}`。

- **`Put` → 两阶段 + 临时文件原子 rename + 路径穿越硬防御**。`handle_put_begin`：
  1. **名字校验**：`validate_entry_name(&name)`（`filetree.rs:1951`，须 `pub(crate)` 化或在 `remote_ws` 复用同等校验）失败 → `PutReady{err: Some(FsErr::PermissionDenied)}`（或新 `FsErr::Io`）。
  2. **子树断言**：`let target = Path::new(&dir).join(&name);` 然后 `dir` 必须 `canonicalize()` 成功且 `target` 的父目录 canonical `starts_with(dir_canonical)`（name 已无 `..`/分隔符，双保险）。不通过 → `PutReady{err}`。
  3. **撞名**：`PutOverwrite::Probe` → `target.try_exists()` 命中回 `PutReady{conflict: Some(PutConflict{is_dir, existing_len})}`，**不写**；`Force/Skip` 时若 `Skip` 直接 `PutResult{Skipped}`，`Force` 建唯一临时文件 `dir/.lumen-put-{req_id}.tmp`（同目录保证 rename 同卷原子）记入 `inflight_put`。
  4. `PutChunk` 按 `seq` 顺序写临时文件、发 `PutChunkAck{seq}`（背压）；`seq` 不连续 → 删临时文件 + `PutResult{err: Io}`。
  5. `PutEnd` → `flush` + `std::fs::rename(tmp, target)`（`Force` 覆盖语义；rename 同卷原子替换）→ `PutResult{Written}`。
- **`MkDir`**：`validate_entry_name(&name)` + 子树断言 + `std::fs::create_dir(dir.join(name))`，`AlreadyExists` 视为成功（幂等）→ `MkDirResult{err: None}`。

被控端**不再维护任何远程展开态** —— `apply_remote_filetree_op`（`main.rs:1063`）整函数删除，#6 自动成立。

## 5. 文件打开流程（#5）

1. 控制端远程树 File 双击 → `RemoteFileTreeOutput.fetch_open = Some(path)`。
2. `mod.rs` 远程分支 → `ShellOutput.remote_fetch_open`（新字段）→ `main.rs` shell_out 消费块（`6795` 附近）：`let req_id = state.remote_ws.next_req_id(); state.remote_ws.start_fetch_open(req_id, path)`。
   - `start_fetch_open` 记在途传输表 `inflight_fetch: HashMap<u64, FetchJob>`（`FetchJob{ kind: FetchKind::Open, file: Option<std::fs::File>, tmp_path: PathBuf, next_seq: u32, started: Instant }`）+ `send_frame(FetchReq{req_id, path})`。
3. `apply_relay`（`remote_ws.rs:846`/`apply` 内 Relay 分支 → 角色门 Controller，照 `remote_ws.rs:790` 附近）：
   - `FileBegin{req_id, total_len}` → 临时路径落地（照 `update.rs:308-311` 清洗模式 + critique 采纳点 9 加 req_id 前缀防撞名）：`std::env::temp_dir().join("lumen_remote_open").join(format!("{req_id}-{safe_basename}"))`，`safe_basename` 把非 `[alnum.-]` 换 `_` 但**保留扩展名**（扩展名通常纯 alnum，默认程序按扩展名匹配）；basename 取被控端给的 `DirEntry.name`（不在控制端做 path 解析）。`create(&tmp_path)` 存句柄；建目录失败/创建失败 → toast + 丢弃。
   - `FileChunk{req_id, seq, data}` → 校验 `seq == job.next_seq`（不连续 → 删临时文件 + 丢弃 + toast），`file.write_all(&data)`，`next_seq += 1`，**发 `FileChunkAck{req_id, seq}`**（背压）。
   - `FileEnd{req_id}` → flush + close（drop File）→ `shell::filetree::open_with_default(&tmp_path)`（`filetree.rs:2077`，含 `reap_in_background`）→ 移除在途表。**finalize 以 FileEnd 为准，不校验 `已收字节==total_len`**（critique 采纳点 7：total_len 仅进度）。
   - `FileErr{req_id, err}` → 删半成品临时文件 + toast（`FsErr` 本地化）。
4. 打开期间树行可显「下载中…」（复用 Loading 风格）；MVP 至少传输完成/失败弹 toast，并在双击即时 toast「正在获取…」（critique I 建议）。
5. **超时/清理**：`pump_remote` 控制端分支扫 `inflight_fetch`，超时项删临时文件 + 移除。会话结束 `clear_remote_filetree`（§8 五处）清空在途表 + 删未完成临时文件。临时目录 `lumen_remote_open` 在 `start()`（`remote_ws.rs:215` 附近）best-effort 整目录清一次（打开后文件可能被外部占用，会话结束删可能失败 → 启动时清更稳）。

**删除** `main.rs:1084` 的被控端 `open_with_default` 分支（随 `apply_remote_filetree_op` 整函数删）。

## 6. 双向传输编排（下载 #7 + 上传）

**统一原语铁律**：被控端只 `ListDir`/`Fetch`/`Put`/`MkDir`，递归全在控制端。

### 6.1 下载（被控端 → 控制端，remote→local）

控制端复制远程文件/文件夹（§7 剪贴板），粘贴到控制端本地目录 `dest_dir`（控制端用 `std::path` 正常解析本地路径）：
- **文件**：`FetchReq` 源 → 边收 `FileChunk` 边写 `dest_dir.join(name)`（`FetchJob.kind = FetchKind::Download{dest: PathBuf}`），ACK 背压同 §5。
- **文件夹**：控制端 DFS，维护**「源 path → 目标本地 path」映射队列**（路径不透明不能控制端拼源端分隔符，但 `entry.name` 是被控端给的安全名，控制端 `parent_dest.join(name)` 本地 join 合法）：
  1. 粘贴时入队根映射 `(src_root → dest_dir.join(src_root_name))`，先 `create_dir_all`。
  2. 对每个目录映射发 `ListDir{path: src}`；`ListDirResult` 回来时对每个 entry 入队 `(entry.path → parent_dest.join(entry.name))`：`is_dir` 则 `create_dir_all` 后再 `ListDir` 递归；文件则 `FetchReq` 写对应本地路径。
  - **环检测**（critique 采纳点 8）：限递归深度（复用 `SEARCH_MAX_DEPTH` 量级常量）+ 已访问 `src.path` 集合，命中即跳过（防 junction 循环）。
- **同名**（写控制端本地）：控制端 `dest.try_exists()` 自己探测 → 命中弹覆盖弹窗（§7）；本地写用 `std::fs` 的 `create_new`（不覆盖）/直接 `create`（覆盖）两路。

### 6.2 上传（控制端 → 被控端，local→remote）

控制端复制本地文件/文件夹，粘贴到远程目录 `remote_dir`（不透明 path，取自远程树节点）：
- **文件**：两阶段 Put —— `PutBegin{dir: remote_dir, name: local_basename, total_len, overwrite: Probe}`（dir+name 分字段，被控端 `Path::join` 自拼，critique 采纳点 3）。被控端回 `PutReady{conflict}` → 无冲突或用户选 `Force` → 控制端读本地文件分块发 `PutChunk`（受 `PutChunkAck` 窗口）→ `PutEnd`。
- **文件夹**：控制端本地 DFS（`std::fs::read_dir`）→ 对每个子目录先 `MkDir{dir: remote_parent, name}` → 递归 → 文件逐个 Put。本地 DFS 同样限深度 + canonical 去环。
- **限流**：递归几百文件不能全并发塞满 WS 出站（无界 `cmd_tx` 内存涨）。控制端限「同时最多 N 个在途 Put」（建议 N=4，§10）；队列串行喂；每个在途 Put 内部再受 `PutChunkAck` 窗口背压。

### 6.3 进度

`FileBegin.total_len`（下载）/ 本地 `metadata().len()`（上传）+ 已传 chunk 累计算单文件百分比；递归按「已完成项/总项」两级。MVP 用 toast「已传输 N 项」（照 `dispatch_delete`→`OpReply`→`drain_ops`→toast 范式 `filetree.rs:625`）。进度条列入 §10 待定。

## 7. 同名覆盖冲突 UX

**文件剪贴板抽象**（挂 `main.rs` State，紧邻既有 `clipboard`，**不挂 `FileTreeState`** —— 跨本地/远程且远程视图下本地树不渲染）：

按钮：`[覆盖] [跳过] [全部覆盖] [全部跳过] [取消]`。危险 `[覆盖]` 用 `Button::fill(pal.error)` + 反相文字（照 `ConfirmDelete` 危险按钮范式）；「全部 X」记 `policy` → 后续同类冲突不再弹（避免 100 文件弹 100 次）。

**消除 TOCTOU**：覆盖决策最终落被控端（上传方向）—— `PutBegin(Probe)` 探测只为「提前弹窗问用户」，真正写入由 `PutBegin(Force)` 用临时文件 + 原子 rename 完成（照 `dispatch_create` 用 `create_new` 防清空旧文件先例 `filetree.rs:597` 附近）；`Skip` 被控端直接 `PutResult{Skipped}`。下载方向（写控制端本地）同理用 `create_new`/覆盖两路。

## 10. 留给海风哥拍板的 UX / 安全决策

**A. 双击目录语义 / cd**
- **推荐**：双击目录 = 控制端本地展开/折叠，**不 cd 被控端**。理由：直接体现 #6（展开纯本地）、修 #4 延迟（Option A 双击=Cd 是慢链根因）、符合文件浏览器直觉（不污染被控端正在跑的 shell）。
- 兼顾：右键菜单留「在被控端 cd 到此」（`remote_cd_dir` 帧，低频显式），留能力不绑死双击。
- 备选：保留双击=Cd（回 A 语义，延迟仍在）。

**B. 粘贴目标入口**
- **推荐**：本地/远程树**右键「粘贴到此目录」**为主入口（复用现成 `dir_context_menu` `filetree.rs:1537` 附近，几乎零新机制；目标明确）。
- 备选：Ctrl+V 粘贴到选中目录/根（需查 main 键盘路由——当前 Ctrl+C 多走终端中断/选区复制，树聚焦态键盘归属有冲突，需确认）。
- 不做：rfd 文件夹选择器（选的是控制端本机，对远程目录无原生选择器，跨侧不通用）。

**C. 覆盖提示样式 + 方向**
- **推荐**：逐文件 + 全部应用混合 `[覆盖][跳过][全部覆盖][全部跳过][取消]`，「全部 X」记 `OverwritePolicy` 套用剩余（避免 100 文件弹 100 次）。危险 `[覆盖]` 红底反相（照 `ConfirmDelete`）。
- **方向**：背景明确要求上传方向必弹；下载方向措辞模糊。**默认两向都弹（一致性）**，待海风哥确认是否下载方向也弹。
- 备选：只「全部覆盖/全部取消」二选一（粒度粗）。

**D. 字节编码 + Fetch 背压实现**
- **编码推荐**：MVP `Vec<u8>` + 64 KiB 块（serde 数字数组膨胀但块小、不引依赖）。带宽是问题再切 base64（+33% vs 3-4×，需加 base64 crate）。
- **Fetch 背压实现**：终稿采「ACK 驱动逐块读」（被控端 worker 领许可读下一块，内存恒为窗口块）。`FETCH_WINDOW=4`、`FILE_CHUNK=64 KiB`、`FETCH_MAX_LEN=2 GiB`，均可调——请海风哥确认这三个数值。

**E. PutBegin 路径拼接**
- **推荐（已并入设计）**：`PutBegin`/`MkDir` 用 `dir: String + name: String` 两字段，被控端 `Path::join` 自拼，控制端不拼跨平台分隔符。

**F. 被控端读/写权限边界（安全，需拍板）**
- **写（Put/MkDir）—— 终稿已定为 MVP 硬约束**：`validate_entry_name`（拒 `..`/分隔符/非法字符）+ canonical 子树断言。**不下放**。
- **读（Fetch）边界**：MVP 推荐「信任已配对控制端」（读任意路径），仅控制字符拒绝 + `FETCH_MAX_LEN`。理由：用户显式配对远控 ≈ 授权对方在本机敲任意命令读任意文件，读权限放宽与现有威胁模型一致。
- 备选（更严）：限「曾 ListDir 列出过的路径」内（需被控端记访问集，复杂）。**请海风哥确认读边界松紧。**

**G. remote→remote 复制粘贴**：本期**不做**（需控制端 Fetch 再 Put 中转，慢/占带宽；或被控端加 cp 原语违反「只读/写少数原语」初衷）。右键菜单按侧禁用，不出现该入口。

**H. 远程「显示隐藏项」开关**
- **推荐**：远程树工具条加「显示隐藏项」勾选（默认关），勾选经 `ListDir.show_hidden` 控制。理由：`.env`/`.gitignore` 等点文件常需下载，本地树过滤隐藏是 UX 选择，远程「文件管理」语义下应可选。本地树维持现状（不过滤变更）。
