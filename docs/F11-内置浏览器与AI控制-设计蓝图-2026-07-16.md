## 0. 结论摘要

- **体积答案：几乎不涨（已实测）。** 选 **wry（系统 WebView2 内核）**：
  浏览器内核用操作系统自带的运行时（Windows = Edge WebView2 Runtime，
  Linux = WebKitGTK 系统包），exe 里只编入 Rust 绑定与面板逻辑。
  **对照实验实测（2026-07-16，见 §5）：wry 全套绑定编进 release exe
  仅 +366KB**，预计集成 Lumen 后 exe 增量 0.4~1MB、安装包（LZMA）
  增量 ~0.2MB，9.1MB → 仍在 9.5MB 以内。
- 反面教材是 **CEF（内嵌完整 Chromium）**：需随包分发 libcef + 资源
  **~150-200MB**，安装包翻十几倍——这才是「加浏览器会变得很大」的方案，
  **否决**。
- **AI 控制通路：** Windows 上 WebView2 就是 Chromium 内核，官方支持开
  **CDP（Chrome DevTools Protocol）调试端口**（微软官方文档明确背书用
  Playwright `connectOverCDP` 驱动 WebView2）。Lumen 只需：开端口（仅
  127.0.0.1、默认关、设置页开关）+ 给 PTY 会话注入发现用环境变量。终端
  里的 AI 用现成的 playwright MCP / chrome-devtools-mcp 即获得导航、截图、
  点击、读 DOM、执行 JS 全套能力，**Lumen 侧零协议开发**。
- **跨平台：** 嵌入层 wry 天然三平台（Windows/macOS/Linux-X11 支持
  `build_as_child`）；AI 控制层 CDP 是 Chromium 私有能力（WebKitGTK/
  WKWebView 无），故控制接口从第一天按平台无关抽象设计，Windows 后端
  = CDP 全功能，Linux/macOS 后端 = wry 通用 API 子集（阶段二）。

## 1. 选型对比（体积问题的完整论证）

| 方案 | 内核来源 | exe/安装包增量 | 与 wgpu/egui 合成 | AI 控制 | 结论 |
|---|---|---|---|---|---|
| **wry 0.55（选定）** | 系统运行时：Win=WebView2、Linux=WebKitGTK、mac=WKWebView | **+1~2MB / +0.5~1MB** | 原生子窗口叠在 wgpu 上（airspace 约束，见 §2） | Win：CDP 官方通路；其它平台走抽象子集 | ✅ 采用 |
| CEF（cef-rs） | 自带完整 Chromium | **+150~200MB** | OSR 离屏渲染可完美合成进 wgpu（可被 egui 遮挡/混合） | CDP 原生 | ❌ 体积不可接受，Rust 绑定生态也不成熟 |
| Servo/verso、blitz 等 | 自带实验引擎 | +20~80MB | 可纹理化 | 无 | ❌ 引擎成熟度远不够真实网页 |
| Ultralight | 自带精简 WebKit | +10~20MB | 纹理化好 | 无 CDP | ❌ 商业授权费；JS 生态残缺 |

**wry 0.55.1 依赖树核实（关键契合点，已亲测）**：Windows 上 wry 依赖
`webview2-com 0.38 + windows 0.61 + windows-core 0.61`——**与 Lumen 现有
`windows`/`windows-sys` 0.61 完全同版**，不引入第二套 Windows 绑定；
`tao`/`winit` 仅是它的 dev-dependency，生产用 `raw-window-handle 0.6`
对接（winit 0.30 原生支持）。注意 **不要**直接用 webview2-com 0.39
（它要 windows 0.62，会与全仓 0.61 冲突拖双份绑定）。

**运行时前提**：WebView2 Evergreen Runtime——Win11 出厂内置、Win10 随
Edge 更新普及（本开发机实测已装 150.0.4078.65）。极老的裸 Win10 缺失时
兜底见 §5。

**诚实代价（不是安装体积而是运行内存）**：WebView2 是多进程浏览器，
打开浏览器 tab 后运行时内存 +100~300MB 量级。设计为**延迟初始化**——
不开浏览器 tab 就零开销。

## 2. UI 形态与嵌入架构

### 2.1 形态：浏览器 tab（侧栏新条目类型）

浏览器作为**侧栏 tab 的新类型**，激活时独占终端工作区矩形；**不做成
终端窗格的一种**（`Tab.panes: Vec<Session>` 的恒非空/PTY 语义不变量
全部不动，侵入最小）。egui 在工作区顶部画一条**浏览器工具栏**（后退/
前进/刷新/地址栏/AI 控制指示/关闭），其下矩形交给 webview。

> 具体挂法（`AppState` 加 `TabKind` 枚举还是独立 `browser_tabs` 列表）
> 由片 1 开工时侦察 main.rs 的 tab 管理代码后定，蓝图不锁死。

### 2.2 嵌入：wry child webview + bounds 同步

- `WebViewBuilder::new().with_url(...).with_bounds(...).build_as_child(&winit_window)`
  —— 官方支持 Windows / macOS / Linux(X11)。
- **bounds 同步**：每帧（或 rect 变化时）把 egui 逻辑矩形 × `scale_factor`
  → 物理像素喂 `set_bounds`；`ScaleFactorChanged` 时重算（DPI 迁移）。
  分屏布局引擎（shell/layout.rs `pane_rects` 同款坐标体系）已是逻辑点，
  换算路径现成。
- **显隐**：切到终端 tab → `set_visible(false)`；切回 → `true`。关闭
  浏览器 tab → drop webview（释放全部浏览器进程）。

### 2.3 airspace 约束（原生子窗口永远盖住 wgpu 内容）

WebView2/WebKitGTK 是原生子窗口，**无法被 egui 绘制内容遮挡**。处置
（Tauri 生态通行做法）：

- 激活浏览器 tab 时如打开**设置页/确认弹窗/右键菜单**等 egui 弹层 →
  `set_visible(false)` 隐藏 webview、露出弹层，关闭弹层恢复。片 3 逐一
  盘点弹层清单做处置矩阵。
- toast（右下角）与浏览器矩形重叠的接受度片 3 实测再定（可选：浏览器
  tab 激活时 toast 改左下）。

### 2.4 焦点与键盘

点击 webview 区域后键盘/IME 由系统直接路由给 webview（不经 Lumen 输入
状态机）；Lumen 全局快捷键在浏览器聚焦时失效是预期行为（Warp/VSCode
内置浏览器同款）。地址栏是 egui 控件、走既有输入路径。边界细节
（Esc 归还焦点、Ctrl+L 聚焦地址栏）片 3 打磨。

## 3. AI 控制通路

### 3.1 阶段一（Windows MVP）：CDP 端口直通

```
Lumen(设置开关开) → WebView2 启动参数 --remote-debugging-port=<port>
                  → PTY 会话注入 env: LUMEN_BROWSER_CDP=http://127.0.0.1:<port>
终端里的 AI       → playwright MCP / chrome-devtools-mcp connectOverCDP
                  → 导航/截图/点击/读DOM/执行JS 全套现成
```

- `ICoreWebView2EnvironmentOptions.AdditionalBrowserArguments` 官方支持
  传 Chromium 开关；微软官方文档「Use Playwright to automate and test
  in WebView2」正是此路（§8 证据 4/5）。wry 侧经
  `WebViewBuilderExtWindows` 的 additional-browser-args API 设置（具体
  签名片 1 开工核对 docs.rs）。
- **端口发现双通道**：①新起 PTY 会话注入环境变量（AI 天然可见，改动
  在既有 spawn 环境注入点）；②写
  `%LOCALAPPDATA%\Lumen\browser-cdp.json`（已在跑的旧会话可读）。
- **免 MCP 备用通路**：`curl http://127.0.0.1:<port>/json` 列页面 target
  → 直连 WebSocket 发 CDP 命令，任何 AI 用 Bash 即可裸控。

### 3.2 阶段二（跨平台）：Lumen browser bridge

CDP 仅 Chromium 有。跨平台抽象：Lumen 起**本地 HTTP JSON API**
（`navigate / back / reload / eval_js / dom_text / screenshot`），
Windows 后端用 CDP 实现全量，Linux/macOS 后端用 wry 通用 API
（`load_url` / `evaluate_script` 等）实现子集。同时它也是给「不想装
MCP 的 AI」的极简通路。IPC 骨架参考 single_instance.rs 的跨平台先例。

### 3.3 「告诉 AI」的三层次接入（海风哥 2026-07-16 拍板采纳）

| 层次 | 机制 | 配置成本 | 排期 |
|---|---|---|---|
| ① 零配置兜底 | PTY 注入 `LUMEN_BROWSER_CDP` 环境变量，用户提示词点名即可（AI 自行 `curl /json` + WebSocket 裸 CDP） | 无 | 片 2 |
| ② 现成 MCP 挂接 | 设置页「AI 控制」旁一键复制接入命令：`claude mcp add chrome-devtools -- npx chrome-devtools-mcp --browser-url http://127.0.0.1:<port>`（或 playwright MCP `connectOverCDP`）——工具清单自动进 AI 上下文，AI 无需提醒即知能控浏览器 | 用户敲一次 | 片 2（E2E 验收走此路） |
| ③ Lumen 自带 MCP | `lumen browser-mcp` 子命令（stdio MCP server，内部 Windows=CDP / Linux=bridge），暴露 `browser_open/screenshot/click/eval/read` 干净工具集；`claude mcp add lumen-browser -- lumen browser-mcp` 一次接入。免第三方依赖 + 跨平台统一 | 用户敲一次 | 片 3/4（与 bridge 同批） |

### 3.4 安全边界

- 端口仅绑 127.0.0.1（Chromium 默认）+ **默认关闭**，设置页开关
  「允许 AI 控制内置浏览器」，开启时浏览器工具栏常驻可见指示。
- 明示文档：本机其它进程同样可连该端口（本地调试协议的固有属性），
  不适合在多人共用机器开启；可选随机端口降低撞见面。
- WebView2 用户数据目录隔离到 `%LOCALAPPDATA%\Lumen\WebView2`（不与
  系统 Edge 混用）。

## 4. 跨平台矩阵（海风哥拍板的硬约束）

| 平台 | 内核 | 嵌入可行性 | AI 控制 | 排期 |
|---|---|---|---|---|
| Windows | WebView2（系统） | `build_as_child(HWND)` 成熟 | CDP 全功能 | **MVP（片1~3）** |
| Linux X11 | WebKitGTK（发行版包） | `build_as_child` 官方支持仅 X11；winit 宿主需 `gtk::init()` + 事件循环内 `gtk::main_iteration_do()` 泵（wry 官方文档路径） | 无 CDP → bridge 子集 | 阶段二（片4） |
| Linux Wayland | 同上 | `build_as_child` 不支持 → XWayland 运行或后续方案，**待研** | 同上 | 阶段二+ |
| macOS | WKWebView | `build_as_child` 支持 | 无 CDP → bridge 子集 | 客户端尚未发 macOS，暂不排 |

- 工程约束：`browser/` 模块从第一天按平台无关接口组织（薄 trait /
  enum dispatch），`#[cfg(windows)]` 后端先行；wry 依赖仅挂
  `[target.'cfg(any(windows, target_os = "linux"))'.dependencies]`，
  **lumen-server 依赖树零污染**（延续 Cargo.toml 既有纪律）。
- Linux 发布物：.deb 增 `Depends: libwebkit2gtk-4.1`；tarball 在 README
  注明运行时依赖。Linux 端本就标注「实验性」，浏览器面板同口径。

## 5. 体积与资源影响（核心问题的量化回答）

**对照实验实测（2026-07-16，scratchpad wry-size-test，release
codegen-units=1 与主仓同配）**：纯 winit 窗口 baseline.exe = 434KB，
winit + wry child webview（真实建出 WebView2）with-wry.exe = 799KB，
**wry 全套绑定增量 = 366KB**；`webview2-com 0.38` 静态链接
WebView2Loader，**target 目录零伴随 DLL**（安装包不需带任何新文件）；
真机冒烟通过（msedgewebview2 子进程拉起、页面加载、退出干净）。

| 项 | 现状 | 加内置浏览器后（wry 方案） |
|---|---|---|
| lumen.exe（release） | — | **+0.4~1MB**（绑定实测 366KB + 面板逻辑；windows 0.61 绑定已在依赖树，feature 并集略增，片 1 出集成后准数） |
| 安装包 Lumen-Setup | 9.1MB | **预计 ≤9.5MB**（366KB 绑定 LZMA 后 ~0.1-0.2MB） |
| 额外分发文件 | — | **无**（WebView2Loader 静态链接，实测无 DLL） |
| 浏览器内核 | — | **0 字节随包**（系统 WebView2 Runtime） |
| 运行时内存 | — | 打开浏览器 tab 后 +100~300MB（多进程浏览器本质；不开 = 零开销，延迟初始化） |

**Runtime 缺失兜底**（极老裸 Win10）：`CreateCoreWebView2Environment`
失败 → 浏览器 tab 显示引导页（一键跳微软 Evergreen Bootstrapper 下载，
~2MB 官方在线安装器，不随 Lumen 分发）；终端主功能零影响。

## 7. 风险清单

- **R1 airspace 遮挡**：egui 弹层被原生子窗口盖住——处置=弹层期隐藏
  webview（业界通行）；残余观感（隐藏瞬间闪烁）片 3 实测调优。
- **R2 焦点/IME 边界**：webview 聚焦时全局快捷键失效、焦点归还路径
  ——片 1 实测定案。
- **R3 CDP 本地端口暴露**：默认关 + 127.0.0.1 + 明示文档 + UI 指示。
- **R4 wry 具体 API 签名**（`build_as_child` / additional-browser-args
  的确切名称与位置）——以 docs.rs wry 0.55 为准，片 1 开工首日核对，
  蓝图按能力面描述不锁签名。
- **R5 Linux Wayland 无 child 嵌入**——阶段二专项调研，不阻塞 Windows
  主线。
- **R6 多 webview 内存**：每 tab 一 webview；共享同一 WebView2
  Environment、限制并存数（如 4）。
- **R7 老 Win10 无 Runtime**：引导页兜底（§5），不挡终端主功能。

## 8. 调研证据

1. wry 0.55.1 crates.io 依赖清单（Windows 目标：webview2-com ^0.38 /
   windows ^0.61 / windows-core ^0.61；tao、winit 仅 dev-dep；
   raw-window-handle ^0.6）——crates.io API 实查。
2. docs.rs/wry：`build_as_child` 支持 macOS/Windows/Linux(X11)；Linux
   非 GTK 宿主（winit）需 `gtk::init()` + `gtk::main_iteration_do()`；
   发行版依赖 webkit2gtk-4.1。
3. webview2-com 0.39.1 依赖 windows ^0.62（与全仓 0.61 冲突，故走 wry
   拖 0.38）——crates.io API 实查。
4. 微软官方《Use Playwright to automate and test in WebView2》
   （learn.microsoft.com/en-us/microsoft-edge/webview2/how-to/playwright
   → playwright.dev/docs/webview2）：官方 CDP 驱动路径。
5. `ICoreWebView2EnvironmentOptions.AdditionalBrowserArguments` 官方
   参考：支持 Chromium 命令行开关（引 chromium.org Run with Flags），
   仅屏蔽 `--user-data-dir` 等少数关键开关。
6. 本开发机实测：WebView2 Evergreen Runtime 150.0.4078.65 已装
   （HKLM EdgeUpdate Clients 注册表 + EdgeWebView\Application 目录）。
7. CEF Windows x64 分发体积 ~150-200MB 为业界经验值（Spotify/CefSharp
   发行物量级），用于对比论证，未做逐版本精查——反正差两个数量级，
   不影响结论。
